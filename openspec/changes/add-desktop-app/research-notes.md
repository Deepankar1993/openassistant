# Desktop App — Research & Expert-Panel Notes

_Generated by a parallel agent team (4 researchers + a 3-member expert panel: Industry Veteran, Devil’s Advocate, Senior Dev). Captured as design-decision provenance for the `add-desktop-app` change._

# Phase 1 — Research

## Codebase Analysis

openAssistant is a single binary crate (package `open-assistant`) with all modules declared in src/main.rs — there is NO src/lib.rs, NO [lib] target, and NO existing Tauri scaffolding (no src-tauri/, no package.json, no tauri.conf.json). The real, working agent loop is src/core/agent.rs `Agent::process(&self, message, ctx: &mut FullContext, session: &mut Session) -> Result<String>`, which builds a system prompt, calls an OpenAI-compatible endpoint via reqwest (config-driven), and does single-shot text-based tool dispatch. CRITICALLY, neither existing UI actually calls this loop: src/ui/web.rs handle_chat and src/ui/tui.rs send_message both return hardcoded "simulated"/"In a full implementation..." placeholder strings. So a Tauri app cannot copy the UIs' wiring — it must wire Agent::process itself (which is straightforward and ready to use). The cleanest path is to add a [lib] target exposing core/config/tools, create a src-tauri/ crate that depends on it, and expose a small set of Tauri commands holding Agent + FullContext + Session behind a Mutex in Tauri's managed State.

**Key findings**

- The crate is binary-only: N:\Projects\rust\openAssistant\Cargo.toml declares package `open-assistant` with no [lib] section, and modules (core, tools, config, ui, etc.) are declared with `mod` in src/main.rs (lines 2-13). A Tauri src-tauri crate cannot `use open_assistant::...` until a library target exists. Recommended: add `[lib] name="open_assistant" path="src/lib.rs"` and a src/lib.rs that re-exports `pub mod core; pub mod config; pub mod tools; pub mod memory; pub mod skills;` (and any others needed), then have main.rs depend on the lib. This is the single biggest structural prerequisite.
- The agent loop to reuse is src/core/agent.rs `Agent::process(&self, message: &str, ctx: &mut FullContext, session: &mut Session) -> anyhow::Result<String>` (lines 55-92). It is fully functional: appends daily note, adds user msg to Session, calls ctx.observe(), builds system prompt, calls `call_llm` (POST {api_base}/chat/completions via reqwest, lines 156-182), runs single-shot tool dispatch in `handle_tool_calls` (lines 184-304), and returns the assistant string. This is the function a Tauri `chat` command must call.
- Agent construction: `Agent::new(model: impl Into<String>)` (agent.rs line 41) builds `default_tools()` and sets workspace_dir to `config::data_dir_default()`. `Agent::with_workspace(dir)` overrides it (line 49). The Tauri command should construct the Agent once from `config::load().await?` model (config/mod.rs ModelConfig.model, default `openrouter/owl-alpha`).
- State to share across commands: a `FullContext` (src/core/persona.rs lines 107-124, `FullContext::new()` gives default Persona+UserModel) and a `Session` (src/core/session.rs lines 19-30, `Session::new(channel, user_id)` or `Session::default()` = channel "cli", user "local"). Both are mutated by process() (ctx is learned-into, session accumulates messages). They MUST persist between calls, so they belong in Tauri managed state behind a tokio Mutex, e.g. `struct AppCore { agent: Agent, ctx: Mutex<FullContext>, session: Mutex<Session> }` registered via `.manage(...)`.
- Existing UIs are STUBS, not references for wiring: src/ui/web.rs `handle_chat` (lines 91-122) returns the literal string 'This is a simulated response. In production, this would run the full 11-step ReAct agent loop.' and never imports Agent. src/ui/tui.rs `send_message` (lines 246-269) likewise returns 'In a full implementation, this would call the 11-step ReAct agent loop...'. The Tauri app should NOT mirror these; it should do what they fail to do and actually invoke Agent::process.
- Reusable UI state shape: src/ui/mod.rs `AppState` (lines 13-65) already models exactly what the desktop UI needs — messages: Vec<Message>, model_name, workspace_dir, permission_mode, total_input/output_tokens, total_cost, plus helpers add_message/clear_messages/update_status. The Message type (src/core/mod.rs lines 27-76) is Serialize/Deserialize with id/role/content/timestamp/metadata, so it can be returned directly to the JS frontend as JSON.
- Config is async and file-based: src/config/mod.rs `config::load()` (line 135) reads ~/.openassistant/config.yaml (auto-creates defaults), `config::save()`, and `config::set(key,value)` (line 158, only a fixed allowlist of keys: model.provider/model/api_key, gateway tokens, security.dm_pairing). data_dir resolves via $HOME/$USERPROFILE + /.openassistant (lines 129-133). Tauri commands for settings should call these. Note api_base default is https://openrouter.ai/api/v1 and api_key starts EMPTY — the desktop onboarding/settings screen must let the user set model.api_key or chat calls will fail (401).
- Tool dispatch is text-based and single-step (agent.rs handle_tool_calls + parse_tool_call line 545, regex `\[TOOL:(\w+):(\{.*?\})\]`). Real tools wired: bash, read, write, edit, glob, grep (via src/tools/*), plus browser/vision/memory/self_manage. Stubbed handlers returning 'In a full implementation...' placeholder text: goal_deliberate (line 323), task/sub-agent (line 370), plan_mode (line 399), perm (line 409). The desktop app inherits real file/bash tools but must treat multi-agent/sub-agent features as non-functional.
- Streaming exists only as a data format, not a live stream: src/core/streaming.rs defines StreamEvent/StreamingSession (NDJSON, Claude-Code stream-json compatible) and `stream_bash_output`, but `call_llm` in agent.rs uses a blocking non-streaming `resp.json()` and Agent::process returns one final String. There is no token-by-token streaming from the LLM today. For a streaming desktop UX you would either (a) emit a single Tauri event when process() returns, or (b) refactor call_llm to use reqwest's streaming + Tauri `Window::emit`/Channel — that is net-new work, not a wrap of existing code.
- main.rs entry points per surface: Tui/Chat -> ui::tui::run_tui() (line 86/219), Web -> ui::web::run_web(port) (line 222). These both spin their own event loops/servers and are NOT reusable as-is inside Tauri (Tauri owns the event loop and webview). The desktop app should bypass ui::tui and ui::web entirely and call core directly.

**Recommendations**

- Add a library target so the core is importable: add `[lib]\nname = "open_assistant"\npath = "src/lib.rs"` to Cargo.toml, create src/lib.rs that does `pub mod core; pub mod config; pub mod tools; pub mod memory; pub mod skills; pub mod ui;` (move the `mod` declarations out of main.rs or have main.rs `use open_assistant::*`). The new src-tauri crate then declares `open-assistant = { path = ".." }` and the existing CLI binary keeps working.
- Create src-tauri/ as a separate crate (cargo-tauri scaffold) depending on the open_assistant lib plus `tauri = { version = "2" }`, `tokio`, `serde`. Do NOT make the existing root crate the Tauri crate, because root is currently a CLI binary with clap; keep that intact and add the desktop app alongside.
- Define managed state: `struct DesktopState { agent: open_assistant::core::agent::Agent, ctx: tokio::sync::Mutex<FullContext>, session: tokio::sync::Mutex<Session> }`. Build it in `tauri::Builder::setup` by loading config, constructing `Agent::new(config.model.model).with_workspace(config.general.data_dir)`, then `.manage(state)`.
- Expose a minimal command surface as #[tauri::command] async fns: `send_message(state, message) -> Result<Message,String>` (locks ctx+session, calls agent.process, returns the assistant Message); `get_status(state) -> StatusResponse` (mirror src/ui/web.rs StatusResponse: model, mode, workspace, message_count, tokens, cost); `get_history(state) -> Vec<Message>`; `clear_conversation(state)` (reset Session to Session::new); `load_config()/save_config(key,value)` wrapping config::load/config::set. Map anyhow::Error to String with `.map_err(|e| e.to_string())` since Tauri commands need Serialize errors.
- Reuse the design tokens already proven in src/ui/web.rs (INDEX_HTML :root vars — --primary #2563eb, --accent #f59e0b, Inter font, card layout, sidebar with Session Info/Usage Stats/Capabilities, message bubbles, typing indicator) as the visual spec for the Tauri frontend so the desktop matches the OpenHumans aesthetic without reinventing it. The HTML/CSS/JS in web.rs is effectively a working reference design to port to the Tauri webview.
- For settings/onboarding, surface model.api_key, model.model, model.api_base in a settings screen wired to config::set/config::save, because the default config ships with an EMPTY api_key (config/mod.rs line 50) — without it, send_message will return an HTTP error from call_llm.
- Defer streaming: ship v1 with non-streaming send_message returning the final Message (matches what Agent::process already does). If a typing-stream UX is required, plan a follow-up to refactor agent.rs call_llm to consume reqwest's SSE stream and push deltas to the frontend via a Tauri Channel/Window::emit — treat this as new development, not reuse.

**Risks**

- Both existing UI surfaces (web.rs, tui.rs) are placeholders that never call the agent — a team member assuming 'just call what the web UI calls' would copy dead stub code. The desktop app must wire Agent::process directly.
- No [lib] target today; without adding one, src-tauri cannot import the core and the whole 'reuse the Rust core' goal is blocked. This is a hard prerequisite, not optional.
- Empty default api_key means the very first chat attempt fails with a provider 401/error unless onboarding/settings is implemented; easy to miss in a happy-path demo.
- call_llm has no error surfacing on bad HTTP status — it does `resp.json()` and pulls choices[0].message.content, defaulting to empty string (agent.rs lines 175-180). API errors (401, rate limit) yield a blank/empty assistant reply rather than a clear error, which will look like a silent failure in the desktop UI. Consider checking resp.status() before parsing.
- Tool dispatch runs real bash/file/edit/write tools with no permission gating actually enforced (perm/plan_mode handlers are stubs returning text). A desktop app exposing send_message gives the model unguarded shell/file access on the user's machine — a security concern that the permission_mode field in AppState only cosmetically tracks.
- Concurrency: Agent::process mutates both ctx and session; if the frontend can fire overlapping send_message calls, you need a single Mutex guarding a process 'turn' (or one Mutex around a combined struct) to avoid interleaved session writes. Locking ctx and session as two separate Mutexes risks deadlock ordering — prefer one Mutex over a combined state struct.
- Agent derives Clone and is cheap, but it loads config inside call_llm on every turn (config::load().await each message) — fine functionally but means runtime config edits take effect mid-session, and adds file I/O per message; acceptable for v1 but worth noting.
- Multi-agent, sub-agent (task), goal deliberation, plan mode, and permission persistence are all stubbed placeholder text in agent.rs — do not advertise these as working desktop features.

- Adopt a left nav rail Chat Memory Integrations Skills Settings pinned at bottom keep the 280px session sidebar inside Chat

- Build a state driven animated mascot owl with idle typing thinking listening talking dreaming states defer a desktop pet overlay

## Features & UX

openAssistant has a rich Rust agent core (persona + evolving user model + dual memory systems + skills + MCP + channels + workflows + text-based tool calling), but the existing UIs are facades: both src/ui/web.rs (/api/chat) and src/ui/tui.rs return hardcoded/simulated responses and never call Agent::process(). No Tauri scaffolding exists yet despite cargo-tauri being installed. The single most important MVP work is therefore wiring a real chat path (Agent::process + streaming call_llm) into a Tauri shell that reuses the Rust core, rather than re-skinning placeholders. The OpenHumans aesthetic (Inter, blue #2563eb, amber #f59e0b, card-based light theme) is documented in openspec/project.md and partially present in the web UI. A modern AI-desktop feature set (streaming chat, history, global hotkey quick-launch, tray, settings UI, memory browser, model/provider switching, light/dark theming, Playwright E2E) maps cleanly onto existing-but-unwired core capabilities; most MVP work is integration plumbing plus Tauri-native shell features (tray, global shortcut), not net-new agent logic.

**Key findings**

- CRITICAL: The Web UI chat handler is a stub. src/ui/web.rs handle_chat() (lines ~91-104) builds a string 'This is a simulated response. In production, this would run the full 11-step ReAct agent loop.' and never invokes Agent::process(). API surface is minimal: only /api/chat, /api/status, /api/clear (web.rs lines 68-73).
- CRITICAL: The TUI is also a stub. src/ui/tui.rs line ~257 builds a `format!` response and sets is_processing; no Agent::new / .process() / call_llm references exist in tui.rs. So neither shipped UI is wired to the real agent.
- The real agent loop exists and is non-streaming. src/core/agent.rs Agent::process() (line 55) does the full flow (daily note, session, ctx.observe, build_system_prompt, call_llm, handle_tool_calls). call_llm() (line 156) POSTs to {api_base}/chat/completions with stream omitted (no 'stream':true) and parses choices[0].message.content in one shot. Real token-by-token streaming to the UI does NOT exist yet for chat.
- A streaming EVENT model exists but is for CLI NDJSON, not UI. src/core/streaming.rs defines StreamEvent (System/StreamEvent/ToolCall/Result/Error) compatible with Claude Code stream-json, but it is not connected to call_llm or any UI transport.
- Model/provider switching is config-only and partial. config::set() (src/config/mod.rs lines 158-172) accepts model.provider, model.model, model.api_key, gateway tokens, security.dm_pairing. Notably model.api_base is NOT settable via set() (must edit YAML). Config is a single flat YAML at ~/.openassistant/config.yaml; there is no notion of multiple saved provider profiles.
- Memory is dual-system and browsable. File memory (src/core/memory.rs MemoryWorkspace: MEMORY.md, memory/YYYY-MM-DD.md daily notes, DREAMS.md) is what the agent reads/writes. SQLite+FTS5 (src/memory/store.rs MemoryStore) exposes store/get/search_fts/list_by_category/delete/count - a ready backend for a memory browser with full-text search. Both are reachable from Rust, so a Tauri command layer can surface them directly.
- Persona + user model are first-class and renderable. FullContext = Persona + UserModel + session stats (src/core/persona.rs), rendered to markdown via build_system_prompt(); UserModel is updated by naive keyword ctx.observe(). These are surfaceable in a settings/'about your assistant' panel but learning quality is shallow.
- No Tauri project scaffolding exists. Grep for tauri/wry across the repo matched only openspec/project.md. There is no src-tauri/, tauri.conf.json, package.json, or frontend build setup yet - so global hotkey, system tray, and windowing are all greenfield (Tauri provides tray + global-shortcut plugins natively).
- No test suite of any kind exists (confirmed by project.md and module scan): no tests/ dir, no #[cfg(test)], so Playwright E2E and Rust unit coverage are both net-new.
- Many advanced core modules are stubs (per project.md and CLAUDE.md): goal_deliberate, task sub-agent return placeholder text; gateway::start_gateway only fully wires WebChat (Discord/Telegram placeholders). MCP, plugins, workflows, channels exist in varying completeness - do not assume end-to-end function for the desktop MVP.

**Recommendations**

- MVP P0 - Wire the real agent before any reskinning. Introduce a thin Tauri command layer (e.g. src-tauri invoke handlers) that calls core::agent::Agent::process() with a shared FullContext + Session, replacing the simulated handlers in web.rs/tui.rs. Until this is done, every UI feature is decorative.
- MVP P0 - Streaming chat UI. Add stream:true support to call_llm (parse SSE 'data:' deltas from the OpenAI-compatible endpoint) and bridge to the frontend via Tauri events (window.emit per token) or a Tauri channel. Reuse the existing StreamEvent enum in src/core/streaming.rs as the wire schema so CLI and desktop share one model. Render token-by-token with a typing indicator and stop button.
- MVP P0 - Conversation history. Persist sessions (src/core/session.rs) to SQLite (reuse rusqlite/MemoryStore patterns) keyed by session id; left-rail list of past conversations with title, timestamp, resume/delete. This also feeds checkpoint and search later.
- MVP P0 - Settings/config UI + model/provider switching. Build a settings panel backed by Tauri commands over config::load/save. IMPORTANT: extend config::set() (or bypass it via direct Config mutation) to include model.api_base, max_tokens, temperature; consider adding named provider profiles (OpenRouter/OpenAI/local) so users can switch endpoints without editing YAML. Mask the api_key field.
- MVP P0 - Theming (light/dark). Implement CSS variables seeded from the documented tokens (Inter, #2563eb, #1d4ed8, amber #f59e0b, success #10b981, border #e2e8f0, bg #f8fafc); add a dark palette and a toggle persisted in config, respecting OS prefers-color-scheme on first run.
- MVP P0 - System tray + global hotkey quick-launch. Use Tauri's tray-icon and global-shortcut plugins: a tray menu (Open / Quick Ask / Quit) and a configurable global hotkey (default e.g. Ctrl+Shift+Space) that toggles a small Raycast-style quick-ask spotlight window that sends one prompt to Agent::process and shows the streamed answer.
- MVP P0 - Memory browser (read-first). Surface MemoryStore.search_fts/list_by_category and MemoryWorkspace (MEMORY.md + daily notes) in a browsable, searchable panel. Read-only is enough for MVP; editing MEMORY.md is P1.
- MVP P0 - Playwright E2E harness. Stand up Playwright against the Tauri webview (or the dev web frontend) with smoke tests: app loads, send message renders streamed reply, switch theme, open settings, open history. Add stable data-testid attributes from day one. Pair with a few Rust #[cfg(test)] unit tests for the new command layer and config round-trip.
- P1 - Tool-call visibility, markdown/code rendering with copy buttons, persona/user-model viewer, editable memory, MCP server status, skills browser (src/skills + skills::check), multi-window, auto-update via Tauri updater, and graceful API-key-missing onboarding (reuse onboarding::wizard).
- P2/Later - Channels/gateway dashboard (Discord/Telegram/Slack) once those move past placeholders; workflow runner UI; checkpoint/restore UI; voice; multi-agent/teams panels. Gate these on the underlying core modules actually being wired end-to-end.
- Cross-cutting - Establish a single 'desktop bridge' module so TUI, Web, and Tauri all funnel through the same Agent::process()/streaming path, honoring project.md's constraint to not fork the agent loop.

**Risks**

- The biggest risk is scope misread: a team could 'reskin' the existing web UI to match OpenHumans and demo it, while it still returns the hardcoded 'simulated response' string - shipping a non-functional app. Wiring Agent::process() must precede UI polish.
- Streaming requires changing call_llm to handle SSE; the current code does resp.json() on a single body. If the chosen provider/model (openrouter/owl-alpha) streams differently or the key is empty (default api_key is blank), chat will silently fail. Need robust error surfacing and an onboarding gate.
- config::set() cannot set model.api_base and there is no multi-profile concept; naive provider switching from the UI may leave api_base pointing at OpenRouter while model/key change, causing confusing failures.
- Tauri 2.x global-shortcut and tray behave differently per OS (Windows is primary here per env, but cross-platform is a stated goal). Global hotkey conflicts and tray quirks are common and need per-platform testing.
- Many core modules (goal_system, subagent task, multi_agent, agent_teams, most of gateway) are stubs returning placeholder text; exposing them in the desktop UI risks presenting non-working features as real. Verify end-to-end wiring before surfacing.
- Playwright against a Tauri (WRY/WebView2 on Windows) target is less standard than against a browser; may need to test the web frontend in a browser for E2E and rely on Rust tests for the native bridge, or use tauri-driver/WebDriver, which adds setup cost.
- No persistence layer for sessions/history exists yet; reusing memory.db naively could conflate conversation history with the FTS memory store - keep schemas separate to avoid polluting memory search.
- Secrets handling: api_key currently lives in plaintext config.yaml; a settings UI that displays/edits it should avoid logging and consider OS keychain later, or at minimum mask it.

# Phase 2 — Expert Panel

## Industry Veteran

**Verdict:** The research is unusually solid and the central insight is correct: this is fundamentally a wiring-and-shell project, not a net-new-agent project. The functional Rust core (Agent::process) exists and works, while BOTH shipped UIs are confirmed dead stubs returning literal "simulated response" strings — so the real risk is the team "reskinning" a facade and demoing a non-functional app. Tauri 2.x in-process over the lib is the right, proven call; the toolchain is already installed and the core links natively with no sidecar. My pragmatic verdict: ruthlessly cut the digest's P0 list. The genuine MVP is four things — (1) lib/bin refactor, (2) one async send_message command wired to Agent::process behind a single Mutex, (3) the existing web.rs HTML/CSS ported into the Tauri webview to match the OpenHumans aesthetic, (4) a settings screen that can set the API key. Streaming, history persistence, global hotkey, tray, memory browser, and the animated mascot are all P1/P2, not P0. The single biggest risk to shipping is NOT technical complexity — it is scope. This team has a digest listing ~9 "P0" items plus a lip-synced animated mascot; that is a quarter of work mislabeled as an MVP, and chasing it will produce a half-finished app instead of a working one. Force the cut.

**Strengths**

- Correct architecture call: Tauri 2.x linking the Rust core in-process (no sidecar/IPC-to-separate-process) is the low-maintenance, proven choice and is exactly how a Rust-core desktop app should be built. cargo-tauri 2.11, Node 22, rustc 1.96 are already present — zero toolchain risk.
- The team correctly identified the load-bearing trap that a careless team would fall into: both web.rs (handle_chat line ~99) and tui.rs (send_message line ~257) return hardcoded placeholder strings and never touch Agent. Verified in source. Naming this explicitly is the most valuable thing in the digest.
- The reuse plan is real, not aspirational: Agent::process(message, ctx, session) -> Result<String> is a clean, synchronous-return seam that maps perfectly onto a single Tauri async command. No refactor of the agent loop itself is required for v1.
- Good instinct to reuse web.rs INDEX_HTML as the literal visual spec rather than reinventing the OpenHumans aesthetic. The CSS variables (#2563eb / #f59e0b / Inter / card layout) already exist and are the design source of truth — porting beats redesigning.
- The concurrency analysis is sound and shows real experience: one Mutex over a combined state struct (not two separate Mutexes for ctx and session) is the correct way to avoid deadlock ordering and interleaved session writes. Honor that.
- Honest about what is fake: goal_deliberate, task/sub-agent, plan_mode, perm, multi-agent, and most of gateway are stubs. The digest repeatedly warns against surfacing them as working features. That discipline is exactly right.

## Senior Dev — Verification Against Latest Trends

**Verdict:** The proposed architecture is fundamentally sound and aligns with current (2025-2026) Tauri 2.x idioms: in-process reuse of the Rust core via a [lib]+bin split, a separate src-tauri crate depending on it, shared managed State holding the Agent loop, async #[tauri::command] handlers, and deferring LLM streaming for v1 are all correct and modern choices. I verified the command/state patterns, capabilities/permissions model, CSP behavior, streaming mechanisms, and E2E testing against Tauri's official v2 docs and Context7. The plan needs three concrete corrections before building: (1) the single-Mutex recommendation must use tokio::sync::Mutex (NOT std::sync::Mutex) because the guard is held across the agent.process() .await — std guard is !Send and will not compile in an async command; (2) Tauri 2 ships CSP OFF by default, so you must explicitly set app.security.csp in tauri.conf.json — and because the Rust core makes the LLM HTTP call (not the webview), you can and should lock CSP to default-src 'self' with NO connect-src to the LLM, which is a security win the digest underplays; (3) the settings/config wiring must bypass config::set() (it silently no-ops unknown keys like model.api_base via a warn-only fallback) and instead mutate the Config struct directly then config::save(). The streaming-deferral is fine, but when streaming lands the modern idiom is tauri::ipc::Channel<StreamEvent> (not per-token Window::emit), and the Playwright testing assumption is the weakest part: official Tauri E2E is WebDriver via tauri-driver (WebdriverIO/Selenium) on Windows+Linux only — macOS has NO native webview driver, so cross-platform E2E parity is a real gap, not just setup cost.

**Strengths**

- In-process reuse via Tauri 2.x in-process (no sidecar) is the correct modern choice given the core is a Rust crate; this matches the OpenHuman reference stack and avoids Electron-style IPC overhead.
- The [lib] + thin CLI bin refactor as a hard prerequisite is accurate and idiomatic - src-tauri cannot `use open_assistant::...` without a library target; this is verified against the actual Cargo.toml (binary-only, mod declarations in main.rs lines 2-13).
- Keeping the existing clap CLI binary intact and adding src-tauri as a SEPARATE crate (rather than converting root to the Tauri crate) is the right call and matches Tauri 2 multi-crate conventions.

The current (2025-2026) idiom for streaming response data (e.g. token deltas, like an HTTP stream) is explicitly tauri::ipc::Channel<T> with a #[serde(rename_all='camelCase')] event enum - this is what Tauri docs now recommend over per-token app.emit for ordered/streaming data. Reusing the existing src/core/streaming.rs StreamEvent enum as the Channel<StreamEvent> payload is the clean path.
- E2E testing: the research leans on Playwright, but official Tauri 2 E2E is WebDriver via tauri-driver with WebdriverIO or Selenium, and it ONLY works on Windows (Edge WebView2 + msedgedriver) and Linux (WebKitWebDriver). macOS has no WKWebView driver, so native cross-platform E2E parity is impossible with the official path.
