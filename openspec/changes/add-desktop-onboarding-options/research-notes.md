# Desktop Onboarding & Options — Research & Expert-Panel Notes

_Generated by a parallel agent team with tiered models (4 Sonnet researchers + a 3-member Opus expert panel: Industry Veteran, Devil’s Advocate, Senior Dev). Design-decision provenance for the `add-desktop-onboarding-options` change._

# Phase 1 — Research

## Terminal-Parity Matrix

Terminal-parity matrix for the openAssistant desktop app. Each terminal capability is assessed for: (a) real vs stub core implementation, (b) exact public Rust API to call from new Tauri commands, and (c) recommended desktop UI surface. Safe-to-surface features are separated from stub-only features that must not be presented as functional.

**Key findings**

- ONBOARDING — Core: REAL (wizard.rs builds and saves a full Config struct via config::save). The wizard is stdin-only (prompt() blocks on io::stdin()). API for desktop: call config::save(&config) directly (src/config/mod.rs:148, async fn save(config: &Config) -> Result<()>) after collecting values in the GUI; there is no library entry point to call — the wizard itself IS the library. Tauri command must replicate the Config construction (model, gateway tokens, security, skills dirs, user_name) and call config::save(). UI surface: a 6-step wizard modal shown on first run when api_key is empty, replacing the current Settings redirect. Steps: (1) data dir, (2) provider/model/api_key, (3) discord+telegram tokens (optional), (4) dm_pairing toggle, (5) user name, (6) summary. Safe to surface: YES — all logic is real I/O, no stubs.
- CONFIG — Core: REAL. API: config::load() -> Result<Config> (src/config/mod.rs:135) and config::save(&Config) -> Result<()> (src/config/mod.rs:148). The config::set() allowlist (src/config/mod.rs:158-172) is deliberately narrow — it only mutates 7 specific keys (model.provider, model.model, model.api_key, gateway.discord_token, gateway.telegram_token, gateway.slack_token, security.dm_pairing). The existing save_config Tauri command already bypasses config::set() correctly by loading the full Config, mutating fields, then calling config::save(). The full Config struct exposes: general.{data_dir, log_level, name, user_name}, model.{provider, model, api_key, api_base}, gateway.{discord_token, telegram_token, slack_token, slack_signing_secret, webhook_port, dm_policy}, memory.{db_path, max_entries, fts_enabled}, skills.{dirs, auto_create}, security.{dm_pairing, allow_from}, vision.{provider, gemini_path}. UI surface: expand the existing Settings view to expose all ConfigDto fields (currently only model/api_base/api_key). Add gateway tokens section. Safe to surface: YES.
- STATUS — Core: REAL (no stubs). API chain: (1) config::load() -> Result<Config> for agent name/provider/model/data_dir. (2) MemoryStore::open_default().await -> Result<MemoryStore>, then store.count() -> Result<i64> for DB entry count (src/memory/store.rs:27 and 190). (3) MemoryWorkspace::from_data_dir(&config.general.data_dir).read_long_term() -> String for MEMORY.md size (src/core/memory.rs:22 and 49). The existing get_status Tauri command already returns StatusResponse {model, provider, mode, workspace, message_count, tools_enabled, api_key_set}. New fields to add: memory_db_entries (MemoryStore::count()), memory_md_chars (ws.read_long_term().len()), data_dir string. UI surface: sidebar Status panel or a dedicated Status view. Safe to surface: YES.
- DOCTOR — Core: REAL (all checks call real code). Diagnostic sequence in run_diagnostics() (src/main.rs:304-343): (1) config::check() -> calls config::load(), (2) MemoryStore::open_default() — tests SQLite open+schema-init, (3) MemoryWorkspace::from_data_dir().init() — creates dirs and MEMORY.md if absent, (4) skills::check() -> SkillEngine::default().count() (returns 0, real but trivial), (5) gateway::check() — checks gateway config (placeholder returns Err unless tokens set), (6) tools::vision::check() — spawns `gemini --skip-trust whoami` process (real external process check). API: all these are direct module function calls; a new run_doctor Tauri command should call each and collect pass/fail per check. Return type: Vec<DiagnosticResult {name: String, ok: bool, message: String}>. UI surface: Doctor tab or Settings sub-section with colored pass/fail rows. Safe to surface: YES for all 6 checks — they are genuine I/O probes, not stubs.
- MEMORY (file workspace) — Core: REAL. All six MemoryWorkspace methods are functional I/O (src/core/memory.rs). API: MemoryWorkspace::from_data_dir(&str) -> Self (line 22); init() -> Result<()> (line 180); read_long_term() -> String (line 49); write_long_term(&str) -> Result<()> (line 74); append_long_term(&str) -> Result<()> (line 57); read_today() -> String (line 85); append_daily(&str) -> Result<()> (line 101); search_files(&str) -> Vec<(String, String)> (line 120). Tauri commands to add: get_memory_md() -> Result<String,String>, write_memory_md(content: String) -> Result<(),String>, append_memory(content: String) -> Result<(),String>, search_memory(query: String) -> Result<Vec<[String;2]>,String>. UI surface: Memory view (currently disabled nav item) — show MEMORY.md in a textarea (read/write), today's daily note (read-only), search box. Safe to surface: YES.
- MEMORY (SQLite/FTS5 store) — Core: REAL with real schema and working FTS5 triggers (src/memory/store.rs). API: MemoryStore::open_default().await -> Result<MemoryStore>; store() / get(key) / search_fts(query, limit) / list_by_category(category, limit) / delete(key) / count(). The MemoryEntry struct is serde-serializable. NOTE: the store is opened per-call (Connection not stored in AppCore state), which is fine for infrequent desktop reads but slightly expensive. Tauri commands to add: search_memory_db(query: String) -> Result<Vec<MemoryEntry>,String> and list_memory_db(category: String) -> Result<Vec<MemoryEntry>,String>. UI surface: Memory view sub-tab for structured entries. Safe to surface: YES.
- SKILLS — Core: REAL for built-in skills and file I/O; real but minimal SkillEngine. API: SkillEngine::load_builtin() -> Result<SkillEngine> (src/skills/engine.rs:87, loads 3 hardcoded skills from skills/*.md via include_str!); load_from_dir(&str) -> Result<usize> (line 124, walks .md files with YAML frontmatter); list() -> &[Skill] (line 234); get(name) -> Option<&Skill> (line 230); count() -> usize (line 242); load_skill_from_file(&PathBuf) -> Result<Skill> (line 154). The Skills CLI command in main.rs (line 173-209) does NOT use SkillEngine — it does raw tokio::fs dir reads. Prefer SkillEngine for the desktop. File create: tokio::fs::write(format!("{}/skills/{}", dir, name), content). Tauri commands to add: list_skills() -> Result<Vec<SkillDto>,String>, read_skill(name: String) -> Result<String,String>, create_skill(name: String, content: String) -> Result<(),String>. UI surface: Skills nav item (currently disabled). Safe to surface: YES for list/read/create. activate_skill() changes tool behavior in-agent — do NOT expose as a UI toggle in this phase (it affects SkillEngine but is not wired into Agent::process).
- AGENTS (sub-agents) — Core: STUB for execution; REAL for definition loading. SubAgentOrchestrator::load_definitions(&str) -> Result<usize> (src/core/subagent.rs:104) walks .claude/agents/*.md and parses YAML frontmatter into SubAgentDef structs — this is REAL. list_definitions() -> Vec<&SubAgentDef> (line 243) is real. BUT spawn() -> Result<String> (line 165) calls execute_subagent() which returns a hardcoded placeholder string (line 276-283): 'Sub-agent NAME completed task: GOAL\nTools used: ...\nSteps: 0/N' — this is a STUB. The sub-agent loop does NOT call the LLM. poll_result() / list_running() are real runtime mechanics around a fake executor. Tauri command to add: list_agents() -> Result<Vec<AgentDto>,String> (safe). Do NOT add a spawn_agent() command — it will silently return fake results. UI surface: Agents view (read-only): show definitions loaded from .claude/agents/ with name/description/tools. Disable any 'Run' button until execute_subagent is wired to a real agent loop. MUST NOT be surfaced as working.
- PLUGINS — Core: REAL for local discovery and enable/disable; marketplace install is a STUB. PluginMarketplace::new(install_dir) + load_installed() (src/core/plugins.rs:74) walks .claude/plugins/*/plugin.json — REAL. list() / list_enabled() / get() / format_status() — REAL. set_enabled() mutates in-memory state only; there is no persistence to plugin.json. install() for PluginSource::Local copies a directory (REAL), PluginSource::Git runs `git clone` (REAL but fragile), PluginSource::Marketplace always returns Err('Marketplace not yet configured') — STUB. Tauri commands to add: list_plugins() -> Result<Vec<PluginDto>,String> (safe), toggle_plugin(name: String, enabled: bool) -> Result<(),String> (note: in-memory only, restarts reset state). Do NOT add an install_from_marketplace() command. UI surface: Integrations nav item (currently disabled) — show plugin list with enable/disable toggles. Add 'Install from local path' dialog only. Label marketplace as 'coming soon'.
- WORKFLOWS — Core: STUB for step execution; REAL for the dependency-resolution scaffolding. WorkflowEngine::register_workflow() / list_workflows() / execute() (src/core/workflows.rs:95) is real in structure: it resolves step dependencies, parallelises ready steps with tokio::spawn, and returns a real duration. BUT each step's closure (line 141-161) just formats a string: 'Step X completed: description' — no LLM call, no real work. built_in_workflows() returns one 'code-review' workflow (line 248). fan_out/fan_in also return placeholder strings. list_runs() always returns an empty Vec (line 236-239). Tauri command to add: list_workflows() -> Result<Vec<WorkflowDto>,String> (safe — shows available workflows with step counts). Do NOT add run_workflow() — it will silently simulate success. UI surface: show workflows read-only in Integrations or a future Workflows section. MUST NOT be surfaced as functional.
- CHECKPOINTS — Core: REAL for create/restore/list in-memory; STUB for persistence. CheckpointStore::new() creates an in-memory Vec<Checkpoint>. create_checkpoint(session_id, description, workspace_dir, file_paths) reads real files, computes real SHA256 hashes, snapshots real content — REAL (src/core/checkpoint.rs:39). restore_checkpoint() writes files back — REAL (line 80). list_checkpoints(session_id) / format_checkpoints(session_id) — REAL (lines 107 and 123). CRITICAL FLAW: CheckpointStore is constructed fresh in each CLI invocation (src/main.rs:284: 'let mut store = CheckpointStore::new()') — all checkpoints are in-memory only and lost on restart. There is no SQLite persistence despite the file comment claiming it. The CLI 'checkpoint --action list' will always return 'No checkpoints for this session' unless checkpoints were created in the same process lifetime. For the desktop app, CheckpointStore can be added to AppCore state (behind the turn Mutex or a separate Mutex) and checkpoints will persist for the app's lifetime. Tauri commands to add: list_checkpoints(session_id: String) -> Result<String,String> (safe if store is in AppCore state). Do NOT imply disk persistence. UI surface: show in chat header as a 'snapshot' button visible during a session only. MUST NOT claim checkpoints survive app restart.
- UPDATE — Core: STUB (trivially prints instructions). CLI handler (src/main.rs:119): 'println!("Use cargo update && cargo build...")'. No real self-update mechanism. Do NOT expose in desktop UI as a button. UI surface: Settings footer showing app version only. MUST NOT be surfaced as a working feature.
- GATEWAY — Core: PARTIAL STUB. gateway::start_gateway() in the CLI starts a WebChat server (partially real). gateway::check() tests whether tokens are configured. Discord/Telegram/Slack are mostly placeholder implementations. The gateway is not relevant to the desktop app directly — it is a separate server process. The Config fields for gateway tokens (discord_token, telegram_token, slack_token) CAN be surfaced in Settings for configuration even though running the gateway requires the CLI. This is the correct architecture. Safe to surface: gateway token fields in Settings/Integrations. Do NOT claim the gateway runs inside the desktop app.

- SAFE TO SURFACE NOW — Onboarding wizard: Implement as a 6-step GUI wizard. The Tauri command should accept all wizard fields and call config::save() directly. Entry point: open_assistant::config::save(&config::Config) at src/config/mod.rs:148. The wizard.rs file (src/onboarding/wizard.rs) is useful as a reference for field order and defaults but must not be called directly (it is stdin-blocking).
- SAFE TO SURFACE NOW — Status view: Add memory_db_entries and memory_md_chars to the existing StatusResponse by calling MemoryStore::open_default().await and MemoryWorkspace::from_data_dir().read_long_term() in the get_status Tauri command. Both calls are non-blocking and already used in the CLI status handler (src/main.rs:102-113).
- SAFE TO SURFACE NOW — Doctor view: Add run_doctor Tauri command that calls all 6 diagnostic checks from src/main.rs:304-343 and returns Vec<{name, ok, message}>. All six checks are real I/O probes. Return structured results so the UI can render colored pass/fail rows.
- SAFE TO SURFACE NOW — Memory view (file): Wire the Memory nav item to a view showing MEMORY.md (textarea, read/write via write_long_term) and today's daily note (read-only). Add search box backed by search_files(). Tauri commands: get_memory_md, write_memory_md, get_today_note, search_memory_files. All call MemoryWorkspace methods from src/core/memory.rs.
- SAFE TO SURFACE NOW — Skills view: Wire the Skills nav item. Use SkillEngine::load_builtin() to list the 3 built-ins, then load_from_dir for user skills at config.skills.dirs[0]. Show each Skill with name/description/category/content. Allow creating new .md files via tokio::fs::write. Tauri commands: list_skills, read_skill, create_skill.
- SAFE TO SURFACE NOW — Agents view (read-only): Wire as a read-only list using SubAgentOrchestrator::load_definitions(&format!('{}/.claude/agents', workspace)) + list_definitions(). Show name, description, tools, model fields. Disable any 'Run' button entirely. Add a note: 'Agent execution is not yet available in the desktop app.'
- SAFE TO SURFACE NOW — Plugins view: Wire as the Integrations nav item. Use PluginMarketplace::new(&format!('{}/.claude/plugins', workspace)).load_installed() + list(). Show name/version/author/enabled status. Toggle via set_enabled() but store the PluginMarketplace in AppCore state (add a Mutex<PluginMarketplace> field) so the enabled state persists within the session. Add local-path install only.
- SAFE TO SURFACE NOW — Settings expansion: Expand ConfigDto to include user_name, data_dir, gateway.{discord_token, telegram_token, slack_token}, and security.dm_pairing. The save_config command already does safe full-Config mutate-and-save.
- DO NOT SURFACE AS WORKING — run_workflow() Tauri command. Workflows appear to execute (the dependency graph runs, steps complete with status=Completed) but zero real work is done. Each step outputs only 'Step X completed: description'. No LLM is called, no files are changed. If added, label workflows as 'preview / coming soon' and only expose list_workflows().
- DO NOT SURFACE AS WORKING — spawn_agent() Tauri command. execute_subagent() in src/core/subagent.rs:267-283 returns a hardcoded string regardless of the model or goal. It does not call the LLM. This will mislead users into believing agent tasks completed successfully.
- DO NOT SURFACE AS WORKING — Checkpoint persistence. The CheckpointStore is in-memory only despite the file comment. Storing it in AppCore state is a valid intermediate solution for within-session checkpoints, but UI must clearly label them as 'session snapshots' and warn that they are cleared when the app closes. Do not implement a 'restore' feature until SQLite persistence is added.
- DO NOT SURFACE AS WORKING — Update button. The CLI Update command just prints cargo instructions. There is no update mechanism. Show version in Settings footer only.
- DO NOT SURFACE AS WORKING — Marketplace plugin install. PluginSource::Marketplace always returns Err. Hide or disable that path in the UI.
- ARCHITECTURE NOTE for new Tauri commands: The current AppCore (src-tauri/src/state.rs) holds only Turn {agent, ctx, session}. To add stateful features (plugins, checkpoints), add new Mutex fields to AppCore: plugins: Mutex<PluginMarketplace> and checkpoints: Mutex<CheckpointStore>. These are initialized in build_core() in src-tauri/src/lib.rs:14. The PluginMarketplace and CheckpointStore are both sync (no async constructors), so block_on is not needed — just call ::new() directly.
- ONBOARDING GATE: The current first-run gate checks api_key_set from get_config. This is correct. The onboarding wizard command should also set general.user_name (currently always 'User' default) — this field is exposed in Config but not in the current ConfigDto. Include it in the wizard and in an expanded ConfigDto.

**Risks**

- CHECKPOINT PERSISTENCE MISMATCH: The CLI checkpoint handler constructs a fresh CheckpointStore on every invocation (src/main.rs:284), so 'checkpoint --action list' always returns empty even if checkpoints were created earlier in the same session. In the desktop app, a CheckpointStore in AppCore state will work within a session but will lose all checkpoints on app exit. This is a significant capability gap vs. what the feature name implies — do not present restore as reliable until SQLite persistence is implemented.
- SUB-AGENT SPAWN SILENTLY SUCCEEDS: SubAgentOrchestrator::spawn() returns Ok(task_id) and sets status to success even though execute_subagent() did nothing real. There is no error returned. A desktop UI that says 'Agent completed successfully' will be actively misleading. The only safe path is a read-only definitions view.
- WORKFLOW STEPS ALWAYS COMPLETE: WorkflowEngine::execute() marks all steps as WorkflowStatus::Completed regardless of whether any real work occurred. The return string 'Workflow code-review completed: 4/4 steps in 0ms' is entirely fabricated. Adding a run_workflow Tauri command without this caveat would mislead users.
- PLUGIN ENABLED STATE IS NOT PERSISTED: PluginMarketplace::set_enabled() mutates in-memory Plugin.enabled but does not write back to plugin.json. App restart resets all toggles to the default_enabled value from the manifest. The UI should warn 'enabled state resets on restart' until persistence is added.
- SKILLS ACTIVATION NOT WIRED: SkillEngine::activate_skill() is implemented and works, but the activated skill's tool restrictions (allowed_tools / disallowed_tools) are not checked anywhere in Agent::process(). Even if the desktop exposes a skill activation button, it will have no effect on the agent's behavior. Do not expose skill activation.
- GATEWAY TOKENS STORED IN PLAIN TEXT: Config is saved as YAML at ~/.openassistant/config.yaml. Discord/Telegram/Slack tokens go in plaintext. The desktop Settings form will be reading and writing these from disk. This is identical to CLI behavior but worth flagging — the desktop should mask token fields exactly as api_key is already masked.
- SKILLS::CHECK() RETURNS 0 NOT REAL COUNT: The skills::check() function called in run_diagnostics() calls SkillEngine::default().count() which returns 0 because default() does not call load_builtin(). The Doctor view will show '0 skills loaded' even when built-ins exist. Use SkillEngine::load_builtin()?.count() instead in the doctor Tauri command for an accurate count of 3.
- CONFIG::SET() ALLOWLIST SILENTLY IGNORES UNKNOWN KEYS: config::set() (src/config/mod.rs:158) has a _ arm that only warns via tracing::warn — it does not return an error. If any code accidentally routes through config::set() for fields like model.api_base or general.user_name, the change will be silently discarded. The existing save_config Tauri command correctly bypasses this, but any future config mutation commands should also bypass config::set() and use the full load/mutate/save pattern.
- MEMORYS TORE NOT IN APPCORE: MemoryStore (SQLite) is opened per-call in the CLI. For the desktop, this means each memory search Tauri command opens and closes the DB connection. This is safe (rusqlite handles it) but wastes ~5ms per call. Consider adding Mutex<Option<MemoryStore>> to AppCore for connection reuse, noting that rusqlite Connection is not Send — wrap in a blocking thread or use a channel.
- VISION CHECK SPAWNS EXTERNAL PROCESS: tools::vision::check() runs `gemini --skip-trust whoami` as a subprocess. On Windows (the current dev platform), this may fail silently if gemini.exe is not on PATH. The Doctor command should catch the Err and display it gracefully rather than panicking.

## Onboarding UX

A concrete 5-step desktop onboarding wizard for openAssistant, grounded in UX best practices from LM Studio, Cursor, Jan.ai, and academic wizard-design research, mapped precisely to the 6-step CLI wizard fields in src/onboarding/wizard.rs. Steps are sequenced by "time-to-value": the minimum blocking information (API connection) comes first; identity and optional integrations are deferred to the end or made skippable; tool/permission consent is a dedicated, non-skippable step immediately before the "Finish" screen. The wizard must be re-triggerable from Settings and the design must match the existing OpenHumans token set (Inter, #2563eb, #f59e0b, #f8fafc, #e2e8f0).

**Key findings**

- WIZARD OVERVIEW — 5 screens (down from CLI's 6 steps): Screen 1 Welcome+Workspace, Screen 2 Provider+Model+API Key, Screen 3 Connection Test (inline, not a separate wizard step but a blocking sub-state inside Screen 2), Screen 4 Tools and Permissions Consent, Screen 5 Identity+Finish. Messaging tokens (Discord/Telegram) and DM pairing are moved to a 'Integrations' view post-onboarding (optional, not blocking chat). Vision/Gemini-CLI check and skills dirs become an async background check shown on the Finish screen.
- CLI WIZARD FIELD MAPPING:
  CLI Step 1 (workspace/data_dir) → Desktop Screen 1: pre-filled with the OS default (~/.openassistant) and a folder-picker button; required, not skippable, but the default is always valid so most users just click Continue.
  CLI Step 2 (provider/model/api_key/api_base) → Desktop Screen 2: the most important screen; provider radio selector (OpenRouter recommended / OpenAI / Custom), model text field (with a smart default per provider), api_base shown only when Custom is selected, api_key password field with show/hide toggle and a masked display. Has an inline 'Test connection' button that fires a real API probe before allowing Continue.
  CLI Step 3 (messaging tokens: Discord, Telegram) → DEFERRED to the Integrations nav view (post-onboarding). Not part of the wizard; too technical and optional for first-run. A brief mention on the Finish screen with a link.
  CLI Step 4 (dm_pairing security) → DEFERRED alongside messaging tokens to Integrations.
  CLI Step 4.5 (user_name) → Desktop Screen 5 (Identity+Finish): the last thing asked before finishing; entirely optional (default 'friend' is used if skipped).
  CLI Step 5 (vision check) → Background async check on Finish screen; result shown as a status pill but does not block the wizard.
  CLI Step 6 (skills dirs) → Not surfaced in the wizard; the default skills dir (~/.openassistant/skills) is written automatically. Advanced users can edit via the Settings view later.
- ESSENTIAL vs OPTIONAL/SKIPPABLE:
  NON-SKIPPABLE (wizard cannot advance without these): api_key field on Screen 2 (must be non-empty to proceed to test), the connection test result (must pass to unlock Continue on Screen 2), and the tool-consent acknowledgement on Screen 4 (user must make an explicit choice — either enable or keep disabled — before finishing). Workspace dir on Screen 1 is pre-filled and valid by default so 'required' is satisfied automatically.
  SKIPPABLE or auto-defaulted: workspace path (use default), model (use provider default), provider (use OpenRouter default), user name (use 'friend'), vision check (background only, non-blocking), skills dirs (use default), all integrations (Discord/Telegram/DM pairing).
- CONNECTION TEST DESIGN (inline in Screen 2, not a full wizard step):
  When the user clicks 'Test connection', the button label changes to 'Testing...' (spinner), the api_key and api_base fields are read-locked, and a minimal /chat/completions request is fired (a single system-prompt-only, zero-token request or a 'ping' style call to /models or /chat/completions with max_tokens=1).
  Three outcomes: (a) SUCCESS — a green checkmark appears, 'Continue' unlocks; (b) AUTH FAILURE (401/403) — red inline error 'Invalid API key. Check the value and try again.'; (c) NETWORK FAILURE (timeout, DNS, 5xx) — amber inline warning 'Could not reach the endpoint. Check the API base URL and your network.' with a 'Try again' link.
  The test result is cached per (api_key, api_base, provider) tuple; editing either field clears the cache and re-locks Continue.
  Under the hood this should call a new Tauri command probe_connection(api_key, api_base, model) -> Result<ProbeResult, String> that fires a real reqwest call (not through Agent::process, to avoid side effects like daily-note writes).
- DATA DIRECTORY PICKER (Screen 1):
  Desktop native file/folder dialog is the right pattern for a desktop app — the CLI uses a plain text prompt, but a Tauri file dialog (tauri-plugin-dialog) gives a native OS picker.
  Show: the current/default path as a styled path-chip with a 'Change folder' button. Clicking opens the OS directory picker.
  Validate: the path must be writable (checked synchronously before Continue is enabled); show an inline red error if not.
  Never show a raw text input for the directory — the OS picker prevents typos and permission confusion.
  The resulting path is written to config.general.data_dir. The wizard should also mkdir -p the path and the skills/ and memory/ subdirs during the Screen 1 → Screen 2 transition so later steps can reliably write to it.
- TOOLS AND PERMISSIONS CONSENT (Screen 4 — non-skippable):
  This is a dedicated, mandatory wizard screen, not buried in Settings. It directly addresses the D10/D13 requirement that tools are OFF by default and the user must make an explicit choice before the desktop ships ungated bash/write access.
  Screen content: a plain-language explanation ('openAssistant can run shell commands and read/write files on your machine to help with tasks. This is powerful but gives the assistant access to your filesystem and terminal.'); two option cards (card A: 'Keep tools disabled — safer, chat only, no file or shell access' [recommended/default]; card B: 'Enable tools — I understand the assistant can run commands and read/write files on my machine'); the user MUST click one card before Continue is enabled.
  If card B is selected, show a secondary confirmation sentence: 'You can change this any time in Settings > Tools.'
  Writes tools_enabled (true/false) to managed DesktopState and persists the choice in config (add a new tools.enabled field or use an existing hook).
  This consent step is the desktop equivalent of the on-first-tool-run confirm dialog, but surfaced proactively so the user understands the posture before they even send the first message.
- IDENTITY + FINISH (Screen 5):
  Single large text input 'What should I call you?' with placeholder 'friend' (matches CLI default). Skippable — a 'Skip' link advances without saving a name.
  Below the name input, show three async background results as status pills (green check or yellow warning): (a) Data dir created and writable, (b) Config saved, (c) Gemini CLI / vision availability (runs crate::tools::vision::check() in the background and updates the pill when done).
  A large 'Start chatting' CTA button navigates to the Chat view and closes the wizard.
  Show a brief 'You can re-run this setup from Settings > Re-run Setup Wizard at any time.' hint at the bottom of the screen.
- PROGRESS INDICATOR:
  A horizontal step bar at the top of the wizard modal/panel showing: 1 Workspace · 2 AI Provider · 3 [inline test] · 4 Permissions · 5 Finish. The step bar uses filled circles (primary #2563eb) for completed steps, the current step is filled with an animated ring, future steps are grey. Step numbers and labels are shown so users know exactly how many screens remain. Research shows this reduces abandonment by communicating determinacy.
- RE-RUNNING ONBOARDING LATER:
  Three entry points must exist: (a) Settings view has a 'Re-run Setup Wizard' button (below the 'About' section); (b) a doctor/status screen shows a 'Setup wizard' link if api_key is empty or config is incomplete; (c) the API-key-empty launch gate (already in the v1 spec) routes directly to Settings, from which the user can click 'Re-run Setup Wizard' to get the full wizard.
  When re-run, the wizard pre-fills all fields from the existing config (api_key shown as '••••...••••' with a 'Change' toggle); the user can step through and change only what they want. The connection test result is cleared and must be re-run if api_key or api_base changed.
  Important: the onboarding wizard should NOT call the CLI wizard.rs run_wizard() function (that function uses stdin/stdout and cannot run inside Tauri's webview). The desktop wizard is a pure frontend flow communicating through the existing Tauri commands (load_config, save_config, probe_connection).
- VALIDATION RULES PER SCREEN:
  Screen 1 (Workspace): path non-empty; path writable (async check on blur or Continue click); error inline next to the path chip.
  Screen 2 (Provider): api_key non-empty (minimum 8 chars recommended, flag shorter but don't block); api_base non-empty and valid URL; model non-empty; connection test must pass.
  Screen 3 (not a screen, inline in Screen 2): see Connection Test Design above.
  Screen 4 (Permissions): one option card must be selected; no other validation.
  Screen 5 (Identity): no required validation; user_name trimmed, max 64 chars, default 'friend' if blank.
  Back navigation: always allowed; all fields retain their values when navigating back (no data loss); connection test result is invalidated if api_key/api_base changed after going back.
- TAURI COMMANDS NEEDED (net-new for the wizard):
  1. probe_connection(api_key: String, api_base: String, model: String) -> Result<ProbeResult, String>: fires a minimal /chat/completions call (max_tokens=1, no system prompt side effects) and returns {ok: bool, latency_ms: u32, error_type: Option<String>}. Must not call Agent::process (to avoid daily-note writes). Implement as a direct reqwest call in a new src-tauri/src/commands/probe.rs.
  2. check_path_writable(path: String) -> Result<bool, String>: creates the dir if absent, checks write permission.
  3. set_tools_enabled(enabled: bool) -> Result<(), String>: already specified as task 2.10 in tasks.md.
  4. The existing load_config / save_config / get_config commands cover reading and writing all other wizard fields.
- EXISTING CLI WIZARD FIELDS NOT NEEDED IN DESKTOP WIZARD:
  Discord bot token, Telegram bot token, Slack token: skip_wizard, surfaced only in the Integrations nav view (post-onboarding, when the gateway module is no longer a placeholder stub — currently most of gateway/discord/telegram are placeholder code per CLAUDE.md).
  DM pairing (security.dm_pairing): same — gateway feature, defer to Integrations.
  Skills dirs (beyond default): not shown in wizard; the default (~/.openassistant/skills) is always created; advanced users can edit via Settings.
  These omissions are intentional: gating first-run on features that are currently placeholder stubs or require separate bot accounts would cause most users to abandon the wizard without ever reaching the chat.

- SCREEN 1 — WORKSPACE (non-skippable, ~30 seconds): Title: 'Where should I keep your data?' Body: 'openAssistant stores your memory, conversations, and settings here. Provider selector: three radio cards with logos/icons (OpenRouter — 'Recommended: 200+ models, one key'; OpenAI — 'OpenAI API'; Custom — 'Any OpenAI-compatible endpoint'). Selecting OpenRouter or OpenAI pre-fills api_base; selecting Custom reveals an editable api_base text field. Upfront consent on the dedicated permissions screen sets expectations cleanly. However, the confirm-before-run dialog (task 4.10) should still exist as a second-layer safety net when tools are enabled — the two mechanisms are complementary, not redundant.

**Risks**

- RISK: tauri-plugin-dialog is not in the current v1 MVP dependency list (tasks.md phase 1 does not include it). Adding it requires a new capability permission entry in src-tauri/capabilities/default.json. If the dialog plugin is deferred, the data directory picker degrades to a plain text input (acceptable fallback), but the native folder-picker is strongly preferred for a first-run UX because it prevents the most common workspace setup error (typo in path).
- RISK: probe_connection is a net-new Tauri command not in the current tasks.md or spec. It must be implemented carefully to avoid Agent::process side effects (daily-note writes, UserModel mutation). Use a raw reqwest::Client call (not Agent::process) with a 1-token /chat/completions payload. This is essential — calling Agent::process for the connection test would write to MEMORY.md on the user's very first interaction.
- RISK: The existing API-key-empty launch gate (spec task 4.2, already implemented) routes to Settings, not to the full onboarding wizard. If the wizard is added as a separate overlay, the two flows (Settings + wizard) must be reconciled so a fresh install routes to the wizard, not just to the Settings panel. The get_config or get_app_state command needs an 'onboarding_complete: bool' flag (or derive it from config completeness) so the frontend can decide which path to take.
- RISK: The step bar shows 4 discrete screens but the connection test is an inline sub-state of Screen 2. If the connection test UI is heavyweight, users may feel Screen 2 is actually two screens. Keep the test affordance compact: a single 'Test connection' button + inline result banner; the progress indicator does not move during the test.
- RISK: Windows-specific data directory default: the CLI wizard.rs uses $HOME/$USERPROFILE + /.openassistant, which on Windows becomes C:\Users\Username\.openassistant (with forward slashes from Rust's path handling). The desktop wizard should show the path with OS-native separators (backslash on Windows). This is a cosmetic risk but can confuse Windows users who see unexpected path formatting.
- RISK: tools.enabled is not a current config.yaml field (src/config/mod.rs does not show a tools section). The set_tools_enabled command writes to DesktopState but may not persist across restarts unless a tools.enabled field is added to the Config struct and config::save() is called. The wizard must persist this choice — losing it on restart would mean users have to re-consent every launch.
- RISK: Gemini CLI vision check in the Finish screen background calls crate::tools::vision::check() which is an async shell invocation. If gemini is not installed, this is fast (binary not found error). If it is installed but slow, the Finish screen pill may spin for several seconds. Add a 200ms timeout to the vision check in the desktop context so the pill resolves quickly with 'Not detected' rather than blocking the screen.

## Options & Information Architecture

Concrete information-architecture recommendation for the openAssistant desktop app, grounded in patterns from Raycast, Msty Studio, LM Studio, Claude desktop, VS Code, and Zed, mapped onto the actual state of the codebase (terminal features, stub status, and already-built Tauri MVP). The IA prioritizes a 5-item nav rail, a sectioned Settings panel, and a staged onboarding flow, with explicit MVP vs. P1/P2 triage and a hard list of terminal features that must not get a panel yet because their cores are stubbed.

**Key findings**

- PATTERN: Narrow nav rail (5-7 items max). Every reference app uses a left icon+label rail at 56-72px wide: Raycast uses a command-palette model without a persistent rail, but its Extensions/Settings open as separate native windows with a left category sidebar (General, Extensions, Advanced, About). Claude desktop uses two columns: conversation list on the left, chat on the right. LM Studio exposes 5 tabs: Discover, Chat, My Models, Local Server, Developer. VS Code Activity Bar has exactly 5 default icons. Zed uses a left panel with project/outline/extensions/collaboration. Msty has collapsible hub sidebars for Persona, Prompts, Library, Knowledge Stacks. Consensus: 5 rail items is the cognitive sweet spot; disable items with a 'soon' pill rather than hiding them (matches current openAssistant implementation, which is already correct).
- PATTERN: Settings is always a categorized left-sidebar + right-content layout, never a flat wall of fields. Raycast Settings: General | Extensions | Shortcuts | Advanced | About. VS Code Settings: Text Editor / Workbench / Features / Extensions / Application (tree, searchable). Zed Settings (2024 redesign): category list on the left, large form on the right, full-text search. LM Studio: Developer tab nests Server Settings + MCP Config + Advanced Load Parameters behind a 'Developer Mode' toggle. Msty: Settings > Data has a 'Danger Zone' section for destructive actions. Universal finding: destructive actions (clear memory, reset config, wipe data) belong in their own clearly labeled sub-section, not inline with regular settings.
- PATTERN: Progressive disclosure via sections, not pages. Best-practice onboarding studies show the first session loses 70-90% of users who hit friction; top-decile apps use 'staged disclosure' — essential info first, then progressive revelation. For openAssistant, the single blocker is the empty api_key (already gated in the current MVP). The wizard.rs CLI flow has 6 steps: workspace, provider/model/key, messaging tokens, security, user identity, tools/vision/skills. A GUI onboarding must collapse this to 3 screens max, with steps 3-6 moved into Settings sections post-setup.
- PATTERN: Separate 'Developer / Advanced' section for power features. LM Studio gates advanced options behind a Developer Mode toggle in Settings. VS Code buries extension-level settings under Application > Developer. Raycast has a dedicated Advanced tab. This is the canonical home for the CLI-level features like checkpoint, workflow, gateway tokens, vision config, and log level — features that power users reach via CLI flags but casual desktop users should never see on first open.
- PATTERN: Diagnostics/health surfaces (Status, Doctor) belong in a dedicated panel, not Settings. Msty exposes data integrity and destructive actions in Settings > Data. But Status and Doctor are read-only observability panels, not configuration. Claude desktop shows session/model info inline in the chat sidebar. LM Studio has a Local Server tab that shows server health. Recommendation for openAssistant: a 'Status' card inside the Chat sidebar (already implemented as the 280px sidebar with model/workspace/message-count) is sufficient for MVP; a dedicated Diagnostics panel is P2.
- CURRENT DESKTOP STATE: The existing frontend/index.html already implements the correct 5-item rail (Chat, Memory, Integrations, Skills, Settings) with Memory/Integrations/Skills disabled as 'soon'. Settings currently has ONE flat form card with 4 fields (model, api_base, api_key, tools toggle). This is correct for MVP but needs sectioning as features land. The onboarding wizard (src/onboarding/wizard.rs) is a 6-step stdin flow that cannot be reused directly in the desktop; its logic must be refactored into Tauri commands.
- STUB STATUS OF CLI FEATURES (verified in codebase): goal_deliberate (agent.rs line 323) returns placeholder text. task/sub-agent (agent.rs line 370) returns placeholder text. plan_mode (agent.rs line 399) returns placeholder text. perm (agent.rs line 409) returns placeholder text. Discord/Telegram gateway (gateway/) is mostly placeholder. Workflow runner dispatches but is not end-to-end wired. Checkpoint is a scaffold. Agents/Plugins commands exist in CLI but dispatch to stubs. Memory (file) is REAL and wired. Skills list/read is REAL. Config load/save is REAL. Status (model, workspace, message count) is REAL. Doctor runs basic checks. These determinations govern which panels get built in each phase.

**Recommendations**

- NAV RAIL (keep current 5-item structure, refine labels and order): 1. Chat (active, wired — keep). 2. Memory (disable with 'soon' — P1 when read-only browser lands). 3. Skills (disable with 'soon' — P1 when skills list/read commands are exposed). 4. Status (rename from nothing — NEW, enable in P1 with real Status+Doctor data). 5. Settings (active, wired — keep). **Risks**

- RISK: 'Integrations' nav item raises false expectations. The current frontend/index.html has Memory, Integrations, Skills, Settings. 'Integrations' strongly implies OAuth/connect flows (Discord bot connect, Telegram link, etc.). The gateway is mostly placeholder. Keeping this label visible — even disabled — invites questions from users and reviewers about when it will be clickable. Mitigation: rename to 'Channels' and keep it disabled, OR remove from rail entirely and move channel token config into Settings > Channels section.
- RISK: Flat Settings form will break badly when sections are added. The current form is a single card. Adding 8 sections to it without refactoring to a two-column Settings view will produce a very long scrolling page with no clear hierarchy. Mitigation: refactor Settings to a two-column layout (category sidebar + content panel) at the same time as the first new section is added (recommended: when Memory settings or General settings land at P1).
- RISK: Onboarding wizard.rs cannot be called from Tauri directly. The wizard uses stdin prompts (std::io::stdin().read_line) which block and have no async boundary. It cannot be invoked from a tauri::command. The wizard's LOGIC (config construction and save) is reusable but must be decomposed into individual save_config calls. Mitigation: the current MVP correctly bypasses wizard.rs and uses save_config directly. Do not attempt to port the wizard as-is.
- RISK: Status nav item without a real diagnostics Tauri command. The recommended Status panel needs at minimum a run_doctor() command returning structured results. Currently doctor runs in the CLI via a synchronous stdout-printing function. It must be refactored to return a structured DoctorReport before it can back a desktop panel. Mitigation: mark Status nav item as disabled until run_doctor Tauri command exists; the Chat sidebar already surfaces the essential live status.
- RISK: Skills and Memory panels depending on commands not yet exposed as Tauri commands. The core modules exist (MemoryWorkspace, SkillManager) but none of their operations are currently exposed as #[tauri::command] functions. Each panel requires new command bridge work. Mitigation: add the commands incrementally per panel; keep nav items disabled until the backing commands land and have at least a config round-trip Rust test.
- RISK: Settings > Channels section showing token fields for placeholder gateways could mislead users into thinking Discord/Telegram integration works. The tokens saved to config.yaml are real YAML writes, but the gateway server that uses them is mostly placeholder. Mitigation: add a visible callout in the Channels section: 'Gateway integration is experimental — the messaging server is not yet fully operational.'
- RISK: Danger Zone section for destructive memory operations is a new capability not in the current CLI (no 'wipe all memory' command exists). Adding it requires implementing new Tauri commands (clear_memory_db, clear_daily_notes, reset_config) that have real, irreversible effects. Mitigation: defer Danger Zone to P2 alongside the Memory browser; do not add it until the memory write path is tested and the confirmation dialog pattern (tool-consent model from spec D10) is established.

## Tauri 2.x Implementation Patterns

**Key findings**

- (1) ONBOARDING — In-app routing is the correct choice, not a separate window. A separate window via WebviewWindowBuilder is possible but creates unnecessary complexity: you need a second window label, a second capability entry (core:webview:allow-create-webview-window), and careful event wiring to close the window and signal the main window on completion. The idiomatic Tauri 2.x approach for a first-run wizard is SPA routing in the existing 'main' window — the frontend detects api_key_set === false (already implemented in the API-key gate at task 4.2) and navigates to an /onboard route. This maps directly to wizard.rs's 6 steps: data_dir, provider/model/api_key, messaging tokens, security, user identity, tools/skills. The save_config command already bypasses config::set() and writes all fields directly; extend it to accept the onboarding payload (GeneralConfig + ModelConfig + SkillsConfig fields) as a single save_onboarding_config command. Emit an 'onboarding-complete' event via app.emit() to trigger the main-window route change if desired. If a separate window is truly wanted (e.g., a narrower 480x600 modal-style window), use: WebviewWindowBuilder::new(&app, 'onboard', WebviewUrl::App('index.html#/onboard'.into())).title('Welcome').inner_size(480.0, 640.0).build()? inside an async command (critical on Windows — synchronous window creation deadlocks due to WebView2). Add 'core:webview:allow-create-webview-window' to capabilities/default.json.
- (2) TEST API CONNECTION — Use tauri-plugin-http which re-exports reqwest. Do NOT add reqwest directly to Cargo.toml; use tauri_plugin_http::reqwest instead. Add to Cargo.toml: tauri-plugin-http = "2". Register in lib.rs: .plugin(tauri_plugin_http::init()). Add to capabilities/default.json the permission 'http:default' scoped to the user-configured api_base (or use a wildcard for flexibility since the endpoint is user-supplied). The command: #[tauri::command] async fn test_connection(state: State<'_, AppCore>) -> Result<String, String> — load config, POST to {api_base}/chat/completions with a minimal 1-token probe payload and the api_key as Bearer auth, check status.is_success(), return Ok('Connected') or Err with status + body. Because the api_base is user-configurable and not known at build time, the capability allow-list must either use a wildcard URL pattern like { "url": "https://*" } or be set dynamically. The simpler approach: add the permission 'http:allow-fetch' explicitly (part of http:default) and use a wildcard scope. Exact capability entry: { "identifier": "http:default", "allow": [{ "url": "https://*" }, { "url": "http://*" }] }. Note: the Rust core already calls the LLM endpoint via its own reqwest (in src/core/agent.rs call_llm) without going through the Tauri HTTP plugin — tauri-plugin-http's permissions only gate frontend JS fetch calls, not Rust-side reqwest. So for a Rust-only test_connection command, you can use reqwest directly (it's already a transitive dependency through open-assistant) without any plugin or capability entry at all. Add reqwest = { version = "0.12", features = ["json"] } to src-tauri/Cargo.toml under [dependencies] and call it directly in the async command.
- (3) FILE/FOLDER PICKERS — Use tauri-plugin-dialog. Add to src-tauri/Cargo.toml: tauri-plugin-dialog = "2". Register in lib.rs: .plugin(tauri_plugin_dialog::init()). The default permissions (dialog:allow-open, dialog:allow-save, dialog:allow-message) are included automatically and cover both file and folder picking — no extra capability entry is required beyond adding the plugin. Add 'dialog:default' to capabilities/default.json to be explicit. Rust API for folder picking: use tauri_plugin_dialog::DialogExt; let folder = app.dialog().file().pick_folder(|path| { ... }); or the blocking variant let folder = app.dialog().file().blocking_pick_folder(); which returns Option<FilePath>. File picking: app.dialog().file().add_filter("YAML", &["yaml", "yml"]).blocking_pick_file(). Expose as Tauri commands: #[tauri::command] async fn pick_data_dir(app: AppHandle) -> Result<Option<String>, String> — call blocking_pick_folder() inside tauri::async_runtime::spawn_blocking to avoid blocking the async executor, then convert FilePath to String via .to_string() or .as_path().map(|p| p.display().to_string()). Folder picking is not supported on Android/iOS but works on all three desktop targets (Windows/macOS/Linux). capabilities/default.json addition: add "dialog:default" to the permissions array.
- (4) OPEN EXTERNAL URLS — Use tauri-plugin-opener (the recommended successor to the v1 shell:open pattern). Add to Cargo.toml: tauri-plugin-opener = "2". Register in lib.rs: .plugin(tauri_plugin_opener::init()). Rust API: use tauri_plugin_opener::OpenerExt; app.opener().open_url("https://openrouter.ai", None::<&str>).map_err(|e| e.to_string())?. JS API: import { openUrl } from '@tauri-apps/plugin-opener'; await openUrl('https://openrouter.ai'). Capability permission: add to capabilities/default.json the entry { "identifier": "opener:allow-open-url", "allow": [{ "url": "https://*" }, { "url": "http://*" }, { "url": "mailto:*" }] } — the allowlist uses glob patterns. The 'https://*' wildcard covers all HTTPS URLs including docs, OpenRouter, provider dashboards. Do NOT use tauri-plugin-shell for URL opening in Tauri 2.x — that plugin is for spawning child processes; tauri-plugin-opener is the correct one for URLs.
- (5) PERSISTING RICHER CONFIG — The existing save_config command already uses the correct pattern: bypass config::set() entirely, load the full Config struct, mutate fields directly, call config::save(). **Risks**

- invoke_handler single-call constraint: if any future developer adds a second .invoke_handler() call (e.g., following a copy-paste example), all previously registered commands silently stop working. Mitigate by adding a comment in lib.rs: // SINGLE invoke_handler — Tauri discards all but the last registration. Add new commands here.
- Windows deadlock on synchronous window creation: WebviewWindowBuilder::build() called from a synchronous Tauri command (non-async fn) deadlocks on Windows due to WebView2's message-pump requirement. All window-creation commands MUST be async fn. The existing commands.rs is already all async — maintain this invariant.
- HTTP capability scope for test_connection via tauri-plugin-http: if the HTTP plugin is initialized, its JS-side permissions (http:default allow-list) scope which URLs the *frontend* can fetch — it does NOT restrict Rust-side reqwest calls. Do not conflate the two. The Rust-side test_connection using reqwest directly needs no capability entry. Only JS invoke('fetch', ...) calls are capability-gated.
- tauri-plugin-dialog blocking_pick_folder on the async tokio executor: calling a blocking function directly in an async command starves the tokio thread pool. Always wrap in spawn_blocking. Failure to do so will cause the app to hang under concurrent command calls.
- FilePath type from tauri-plugin-dialog: the return type is tauri_plugin_dialog::FilePath, not std::path::PathBuf. On desktop it wraps a PathBuf; on mobile it wraps a URI. Call .as_path() to get Option<&Path> and convert with display().to_string(). Do not call .to_string() directly as that serializes the enum variant.
- Config round-trip with skills.dirs as Vec<String>: serde_yaml serializes Vec<String> as a YAML sequence. If the onboarding wizard sends a comma-separated string from the frontend (matching the wizard.rs stdin behavior), split it server-side before writing to skills.dirs — do not store the raw comma-separated string in the field.
- opener:allow-open-url wildcard https://*: this allows opening any HTTPS URL, including potentially malicious ones if the frontend were compromised. For a locally-served app with strict CSP this is acceptable risk. If the app ever loads remote content (e.g., in an iframe or via remote webview URL), narrow the allow-list to specific provider domains.
- The config::set() allowlist silently no-ops unknown keys (tracing::warn arm) — this is already handled correctly by the commands.rs save_config bypass. Risk: any future developer who calls config::set() from a new command for a field not in the match arm will get a silent no-op with only a log warning.
